# Consultation Brain: Business Component Design Principles

To seamlessly bridge the gap between "Consultant-Client UX" and "Systematic Engineering Principles," we have evolved our design philosophy from "passive support tools" into an "intelligent framework for collaborative dialogue."

These principles transform technical engineering requirements into professional business axioms.

## 🔄 Business Components Design Principles

### 1. `LogicConflictAlert.vue`: The Consensus Validator

* **Design Principle:** **Calibration Guidance**
* **Core Philosophy:** When the system detects a conflict between the dialogue and established boundaries (BOUNDARY), avoid cold "error alerts." Use "differentiated guidance" to treat conflicts as opportunities for information synchronization. Through high-contrast visual tokens (Amber/Red), we guide the user to calibrate, transforming "logic flaws" into "moments for alignment."

### 2. `ExtractionTagStream.vue`: The Consensus Stream

* **Design Principle:** **Node Extraction**
* **Core Philosophy:** This is more than information display; it is the process of mapping linear dialogue into knowledge graph nodes (ENTITY, EVENT, etc.). The design focuses on "Tag Actionability," ensuring every consensus extracted from the conversation can be converted into a structured graph object with a single click, reducing the cognitive load of manual documentation.

### 3. `ConsensusProposalList.vue`: The Consensus Anchor

* **Design Principle:** **Consensus Defense**
* **Core Philosophy:** Consensus in a consultation is fragile. The design focuses on "State Locking." Each proposal/consensus has a clear lifecycle (Proposed -> Review -> Locked). Visual cues emphasize "permanence," ensuring the client perceives progress and develops psychological reliance and security in the established goals.

### 4. `BlindSpotNavigator.vue`: Global Navigator

* **Design Principle:** **Dimensional Completion**
* **Core Philosophy:** Avoid interruptive warnings. Instead, use structural detection to provide "suggestive prompts" when the dialogue network density is insufficient (lacking CONCEPT or RESOURCE). The goal is to assist both parties in elevating from localized anxiety to a strategic, holistic perspective, preventing them from falling into "blind spot loops."

### 5. `StrategicBaselineCard.vue`: The Strategic Cornerstone

* **Design Principle:** **Stability Visualization**
* **Core Philosophy:** Strategic baselines must possess undeniable visual authority. Through distinct hierarchical grouping and border tokens, the component highlights its status as "Validated Decisions." It does more than show data; it symbolizes high-level agreements between consultant and client, stabilizing emotional fluctuations during the dialogue.

### 6. `ReflectionLogSandbox.vue`: The Human Reflection Sandbox

* **Design Principle:** **Low-Friction Introspection**
* **Core Philosophy:** Acts as a "mental offloading slot" during consultation. The design pursues ultimate input velocity, ensuring that when an idea sparks, the user can immediately cache it through category tabs (INSIGHT vs. QUERY). These insights are raw material for graph generation; the component is designed to be "always available, never intrusive."

---

## 💡 Summary: Advantages of the Componentized Architecture

By adopting these principles, the system transforms from "technical engineering" into "business logic":

* **Collaborative Empathy:** All components possess a design logic understood by both the consultant and the client.
* **Structural Readiness:** Every component performs pre-processing for the backend's `NodeType` and `EdgeType`, ensuring that once the system enters "Knowledge Graph Mode," the data is already highly structured.
* **Extensibility & Stability:** This methodology—mapping dialogue behaviors to specific data structures—ensures that the system maintains high consistency and professional integrity as we expand to include more specialized tools in the future.
